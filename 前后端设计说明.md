# 前后端设计说明

> 维护约定：本文件是当前项目的设计基线。以后每次修改前端、后端、接口、数据结构、部署方式、核心机制时，都必须同步更新本文件。

## 1. 项目定位

本项目是一个“虚拟地球 Online”人生模拟游戏原型。

当前目标不是做传统数值 RPG，而是做一个“玩家设定人生 + AI/NPC 推理 + 世界状态推进 + 社会机制反馈”的人生体验系统。

当前已具备的基础能力：

- 用户注册、登录、鉴权
- Persona 人格标定与绑定
- Dify Chatflow 驱动的 NPC 对话能力
- NPC 消息聊天与主动通知
- 前端游戏场景化 UI
- 世界状态面板 v1
- 主剧情分支与剧情过场能力 v1

## 2. 总体架构

项目由两部分组成：

- 前端：`Vite + React + React Router + Zustand`
- 后端：`FastAPI + SQLAlchemy + PostgreSQL + APScheduler`

职责划分：

- 前端负责游戏场景、交互面板、剧情演出、设备 UI、消息展示、世界状态呈现。
- 后端负责鉴权、Persona 存储、NPC 消息生成、主动推送调度、剧情状态推进、数据持久化。

- `StoryScene`

### 4.3 剧情系统设计

当前主剧情系统分两层：

1. 剧情分支层

- 校园转场线 `campus_transition`
- 都市打工线 `urban_pressure`
- 家庭牵引线 `family_obligation`
- 职业突破线 `career_breakthrough`

2. 剧情场景层

每条分支包含多个 scene，每个 scene 包含：

- `title`
- `scene_type`
- `lines`
- `choices`
- `background`
- `music_cue`
- `enter_effect`

剧情推进机制：

- 用户首次进入时先拿到可选主线
- 选择主线后写入 `StoryProgress`
- 当前 scene 通过 `current_scene_key` 表示
- 用户选择剧情选项后，推进到下一个 scene
- 选择结果会记录到 `flags`

当前 `flags` 中主要记录：

- `started`
- `branch_started_at`
- `last_choice_key`
- `last_choice_effect_key`
- `last_choice_at`
- `scene_history`

## 5. 当前主动消息推送机制梳理

这是你当前后端“随机推送”的真实设计机制。

### 5.1 入口

主动消息主要由这几个入口触发：

- 用户进入场景时，前端主动调用 `POST /messages/trigger/random`
- 用户打开手机时，前端再次调用 `POST /messages/trigger/random`
- 后端定时任务调用 `run_random_push_cycle`
- 用户轮询消息时，后端也会顺带处理应投递消息

### 5.2 调度条件

Persona 上当前有这几个字段控制主动推送：

- `npc_push_enabled`
- `npc_push_frequency_minutes`
- `next_npc_push_at`

后端调度流程：

1. APScheduler 定时运行
2. 找出所有 `next_npc_push_at <= now` 的 Persona
3. 对每个用户执行 `generate_random_message_for_user`

### 5.3 当前“随机”的本质

它并不是完全无逻辑的随机文本，而是“弱状态感 + 时间桶 + 母亲 NPC 模板化触发”。

当前 `generate_random_message_for_user` 的逻辑大致是：

1. 获取当前用户 Persona
2. 获取当前 NPC 会话 `NpcConversationSession`
3. 检查最近是否刚发过消息，避免过近重复推送
4. 根据时间和最近聊天情况，解析一个 trigger
5. 使用这个 trigger 构造 Dify 输入
6. 由 Dify Chatflow 生成母亲 NPC 的主动话术
7. 写入 notification/chat 两个消息通道
8. 重新安排下一次推送时间

### 5.4 当前 trigger 来源

当前主动推送 trigger 主要来自 `_resolve_mother_trigger()`：

- `morning_greeting`
- `late_night_check`
- `job_follow_up`
- `money_concern`
- `weather_care`
- `holiday_check_in`

它的判断依据主要是：

- 当前小时
- 距离上次聊天时间
- 一周中的天数桶

所以当前推送“看起来随机”的根本原因是：

- 没有真正读取玩家的世界状态
- 没有基于事件紧急度决定消息
- 没有将剧情进度作为主要触发器
- 当前只有一个测试 NPC：母亲
- Dify 负责润色文本，但不负责决定“现在到底该说什么才重要”

### 5.5 当前版本已补充的改进

目前已经把剧情进度提示接入主动推送上下文：

- 若玩家已进入主线，后端会把 `story_branch / current_scene / last_choice_effect` 注入 NPC 主动消息上下文

这意味着现在的主动推送已经开始具备“剧情感知”，但还不是真正的“社会机制驱动”。

## 6. 当前世界模拟机制 v1

当前世界模拟已经开始从前端本地原型，迁移为后端驱动。

### 6.1 已有状态维度

- 金钱：余额、月收入、月支出、现金压力
- 健康：精力、睡眠、身体状态
- 情绪：稳定度、焦虑、孤独
- 关系：母亲、朋友、工作、机构
- 时间：日期标签、截止提醒
- 事件：待处理现实事件
- 行动：玩家可执行动作
- 日志：决策结果记录

当前已落到后端的数据模型：

- `PlayerWorldState`
- `PlayerWorldEvent`

### 6.2 已有动作机制

当前动作会通过后端 `/world/action` 修改状态，例如：

- 处理账单
- 早睡恢复
- 给母亲回消息
- 保持沉默
- 通勤加班
- 出去散步
- 认真沟通
- 接住机会

### 6.3 当前已接入的状态驱动能力

- 世界状态由后端持久化
- 世界事件由后端生成与维护
- 主剧情选择可写入世界状态效果
- 主动推送已开始读取“剧情进度 + 世界事件提示”

### 6.4 当前仍未完成的问题

- 世界状态还未按天推进
- 事件引擎还只有第一层规则，不够复杂
- 关系网络仍然过于单薄
- NPC 仍然主要只有母亲一个
- 前端虽然已偏游戏活动页风格，但还没完整实现你目标中的多层分镜式 UI

## 7. Docker 部署与启动

后端默认通过 Docker 方式部署与启动，这也是当前推荐方式。

### 7.1 启动方式

```bash
cd earthMockServer
docker compose up -d --build
```

### 7.2 常用检查命令

```bash
docker compose ps
docker compose logs -f app
docker compose logs -f postgres
```

### 7.3 停止服务

```bash
docker compose down
```

### 7.4 说明

- `app` 为 FastAPI 服务
- `postgres` 为 PostgreSQL 数据库
- 推荐以后继续沿用 Docker 维护后端环境一致性

如果仅用于本地开发调试，也可以直接运行：

```bash
cd earthMockServer
uvicorn app.main:app --reload
```

但正式联调和部署优先使用 Docker Compose。

## 8. 接下来要完善的核心方向

为了把它真正做成“模拟真实社会运行机制”的人生游戏，建议按下面顺序推进。

### 8.1 第一阶段：把世界状态后端化

优先新增后端持久化模型：

- `player_world_state`
- `player_event`
- `player_action_log`
- `player_schedule`

目标：

- 所有金钱、健康、情绪、关系、时间都由后端保存
- 前端只负责显示，不再自己维护真状态

### 8.2 第二阶段：建立事件引擎

把现实事件从“写死在前端”变成“后端根据状态生成”。

事件类型建议优先做：

- 房租/账单
- 睡眠透支
- 家庭压力
- 求职/工作绩效
- 社交疏离
- 偶发机会

每个事件都需要：

- 触发条件
- 紧急度
- 截止时间
- 可选动作
- 立即后果
- 延迟后果

### 8.3 第三阶段：把主动推送改为状态驱动

主动推送不应该再以“时间桶随机关心”作为主体，而应变成：

- 有事件时才推
- 推送优先级由世界状态决定
- NPC 只是事件表达者，不是事件决策者

理想链路：

`世界状态 -> 事件引擎 -> 推送路由 -> NPC 叙事润色 -> 通知/聊天/剧情入口`

### 8.4 第四阶段：扩展 NPC 社会网络

当前只有“虚拟母亲”一个 NPC，适合测试，不适合构建社会模拟。

下一批建议 NPC：

- 父亲
- 朋友
- 同事/上司
- 房东
- 恋爱对象或潜在人际对象
- 学校/机构/平台型系统角色

### 8.5 第五阶段：前端升级为“人生操作台”

前端可继续扩展为以下模块：

- 世界状态总览面板
- 时间推进器
- 现实事件收件箱
- 剧情过场播放器
- 关系网络面板
- 金钱流动面板
- 健康/情绪波动面板
- 重要人生节点回放

## 9. 当前设计结论

当前项目已经从“AI 聊天壳子”开始过渡到“人生模拟游戏框架”，但离“真实社会模拟”还有关键一步：

要把核心驱动力从“随机聊天”切换为“状态、事件、关系、时间、资源”。

也就是：

- AI 不负责决定世界
- 世界机制先决定发生什么
- AI 负责把这些事说得像真实的人

这是后续整个系统架构最重要的原则。
