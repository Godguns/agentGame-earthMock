# 前后端设计说明

> 维护约定：本文件是当前项目的设计基线。以后每次修改前端、后端、接口、数据结构、部署方式、核心机制时，都必须同步更新本文件。

## 1. 项目定位

本项目是一个“虚拟地球 Online”人生模拟游戏原型。

当前目标不是做传统数值 RPG，而是做一个“玩家设定人生 + AI/NPC 推理 + 世界状态推进 + 社会机制反馈”的人生体验系统。

当前已具备的基础能力：

- 用户注册、登录、鉴权
- Persona 人格标定与绑定
- Dify Chatflow 驱动的 NPC 对话能力
- NPC 消息聊天与主动通知
- 前端游戏场景化 UI
- 世界状态面板 v1
- 主剧情分支与剧情过场能力 v1

## 2. 总体架构

项目由两部分组成：

- 前端：`Vite + React + React Router + Zustand`
- 后端：`FastAPI + SQLAlchemy + PostgreSQL + APScheduler`

职责划分：

- 前端负责游戏场景、交互面板、剧情演出、设备 UI、消息展示、世界状态呈现。
- 后端负责鉴权、Persona 存储、NPC 消息生成、主动推送调度、剧情状态推进、数据持久化。

**虚拟形象素材规范：**

- 形象图片路径：`/assets/avatars/{avatar-id}.png`（如 `/assets/avatars/avatar-01.png`）
- 放置位置：`public/assets/avatars/`
- AvatarSetupScreen 和 WorldFieldScreen 中的角色展示位置均预设了 CSS 渐变色占位，替换为 `<img>` 标签即可接入真实素材
- 建筑/场景素材预留路径：`/assets/world/{scene-key}/{building-id}.png`

- 主题规范补充：
  设置中心的卡片激活态统一为白底、黑字；手机联系人与聊天界面在白天主题下使用深色文字与深色头像边框，在夜间主题下切换为浅色文字与浅色头像边框。
- 手机聊天细则补充：
  白天主题下，聊天页的状态栏、返回按钮、时间戳、输入提示、placeholder、联系人头像文字都必须保持深色可读；仅“玩家本人发送的气泡”允许继续使用浅色按钮式前景。
- 世界状态面板补充：
  世界状态面板避免使用整块纯黑重卡，外层允许深色，但内部事件卡、行动卡、图表底盘必须保留层次差与轻微发光；四个环形指标分别使用资金橙、健康绿、情绪蓝、关系粉，并且中心数值优先展示真实关键值而不是统一小字号百分比。
- 世界行动规则补充：
  “处理账单”必须和活跃账单事件绑定，没有 `rent_due` 活跃事件时不可继续重复扣钱；行动卡需要根据当前事件上下文显示可用/不可用与语义提示，避免玩家误触造成负反馈。
- 剧情推进规则补充：
  剧情场景如果选择后不存在 `next_scene_key`，应视为该段剧情完成，后端必须清空 `current_scene_key`，前端据此退出当前场景态，避免剧情面板停留在旧场景无法继续。
- 打包策略补充：
  前端构建采用按依赖域拆包，至少拆分为 `React 基础`、`路由`、`状态管理`、`动效`、`3D/后处理` 等独立 chunk，避免手机壳、消息系统与 3D 背景全部挤进单一主包。
  场景背景类重资源组件应优先懒加载，避免阻塞主界面与手机面板的首屏渲染。
- 交互目标：避免传统后台面板感，更接近”可操作的生活模拟游戏界面”
- 当前已改动：
  - 世界状态图表由条形改为圆形
  - 新增剧情过场层
  - 面板向更强的游戏 HUD 风格演进

## 5. 当前主动消息推送机制梳理

这是你当前后端“随机推送”的真实设计机制。

### 5.1 入口

主动消息主要由这几个入口触发：

- 用户进入场景时，前端主动调用 `POST /messages/trigger/random`
- 用户打开手机时，前端再次调用 `POST /messages/trigger/random`
- 后端定时任务调用 `run_random_push_cycle`
- 用户轮询消息时，后端也会顺带处理应投递消息

### 5.2 调度条件

Persona 上当前有这几个字段控制主动推送：

- `npc_push_enabled`
- `npc_push_frequency_minutes`
- `next_npc_push_at`

后端调度流程：

1. APScheduler 定时运行
2. 找出所有 `next_npc_push_at <= now` 的 Persona
3. 对每个用户执行 `generate_random_message_for_user`

### 5.3 当前“随机”的本质

它并不是完全无逻辑的随机文本，而是“弱状态感 + 时间桶 + 母亲 NPC 模板化触发”。

当前 `generate_random_message_for_user` 的逻辑大致是：

1. 获取当前用户 Persona
2. 获取当前 NPC 会话 `NpcConversationSession`
3. 检查最近是否刚发过消息，避免过近重复推送
4. 根据时间和最近聊天情况，解析一个 trigger
5. 使用这个 trigger 构造 Dify 输入
6. 由 Dify Chatflow 生成母亲 NPC 的主动话术
7. 写入 notification/chat 两个消息通道
8. 重新安排下一次推送时间

### 5.4 当前 trigger 来源

当前主动推送 trigger 主要来自 `_resolve_mother_trigger()`：

- `morning_greeting`
- `late_night_check`
- `job_follow_up`
- `money_concern`
- `weather_care`
- `holiday_check_in`

它的判断依据主要是：

- 当前小时
- 距离上次聊天时间
- 一周中的天数桶

所以当前推送“看起来随机”的根本原因是：

- 没有真正读取玩家的世界状态
- 没有基于事件紧急度决定消息
- 没有将剧情进度作为主要触发器
- 当前只有一个测试 NPC：母亲
- Dify 负责润色文本，但不负责决定“现在到底该说什么才重要”

### 5.5 当前版本已补充的改进

目前已经把剧情进度提示接入主动推送上下文：

- 若玩家已进入主线，后端会把 `story_branch / current_scene / last_choice_effect` 注入 NPC 主动消息上下文

这意味着现在的主动推送已经开始具备“剧情感知”，但还不是真正的“社会机制驱动”。

### 5.6 当前版本新增的节流规则

为了节省 token，并避免 NPC 在玩家长期不回应时持续打扰，当前已新增一条主动推送节流规则：

- 同一个 NPC 在同一天内，主动推送 3 条消息后，如果玩家始终没有回复
- 系统会额外发送 1 条“记得回复我”的留言提醒
- 这条提醒发送后，该 NPC 当天不再继续主动推送
- 玩家只要回复一次，当天计数立即清零，主动推送恢复
- 跨天后计数也会自动重置

当前这条规则通过 `NpcConversationSession.meta` 维护，不额外新增表结构。

## 8. 近期前端交互规范补充

### 8.3 玩家虚拟形象初始化页

路由：`/avatar/setup`

**功能：**

- 提供 12 个可选的虚拟形象（晨海、白露、时雨、夏澄、鸢尾、凛夜、铃音、深月、栀夏、璃光、真昼、雾羽），每个形象有名字、气质标签（tag）、主题色（accent）
- 玩家最多可选择 6 个形象加入已选列表，点击 `+/-` 按钮切换选中状态
- 左侧 Hero 区域展示当前聚焦形象的预览卡片和元信息
- 右侧网格展示全部 12 个形象的角色池，支持聚焦高亮
- 底部已选条（selected strip）展示当前已选的 6 个槽位（已填充/空槽）
- **确认流程**：点击「确认选择」后，选择结果持久化到 localStorage，并自动跳转到 `/world/field`
- **预览模态框**：展示所有已选形象的大尺寸预览
- 确认后有白色闪烁过场动画

**素材预留：**

每个形象的图片路径预设为 `/assets/avatars/{avatar-id}.png`，当前使用 CSS 渐变色占位。后续接入真实像素风角色素材后，只需：
1. 将图片放入 `public/assets/avatars/` 目录
2. 替换 JSX 中的占位 `<span>` 为 `<img>` 标签即可

**关联：**
- 选择结果存储在 `localStorage` 的 `earth-online-avatar-selection` 键中
- WorldFieldScreen 读取该值显示玩家在地图上的角色

### 8.4 第二场景页（NIKKE 风格世界地图）

路由：`/world/field`

**功能：**

- **人格驱动场景选择**：根据玩家 Persona 中设定的城市自动匹配默认场景：
  - 一线城市（北京/上海/广州/深圳等）→ metro
  - 县城/乡村 → village
  - 其他 → town（城镇）
- **三个场景类型**：metro（一线城市）、town（城镇）、village（乡村），各有不同色调和氛围
- **类 3D 视差背景**：
  - 天空层（sky）：渐变氛围光
  - 远层（far）：城市/乡村剪影轮廓
  - 中景层（mid）：地面平面
  - 前景层（front）：透视网格地面 + 交互建筑点
- **建筑交互点**：每个场景 4 个建筑/地标，以图标+标签的浮动按钮形式分布在地图上：
  - metro：超市、公司、咖啡店、地铁站
  - town：学校、便利店、公交站、诊所
  - village：耕地、院子、山路、小卖部
- **角色行走系统**：
  - 点击建筑 → 角色从当前位置平滑移动到目标建筑
  - 移动时长根据距离动态计算（400ms-1200ms）
  - 行走时有上下摆动动画（avatar-bob），到达时有落地动画（avatar-land）
  - 角色阴影随行走状态变化
- **到达触发剧情**：角色到达目标建筑后，自动弹出 VN 风格剧情对话框
- **剧情面板（VN 风格）**：
  - 底部滑入式面板，半透明毛玻璃背景
  - 展示建筑信息、场景描述、叙事文本
  - 提供「进入互动」「观察周围」「离开」三个选择按钮
  - 持续提示指示器（呼吸动画）

**交互流程：**
```
点击建筑 → 建筑高亮 → 角色行走 → 到达 → 剧情弹出 → 选择互动
```

**HUD 层：**
- 左下角显示当前场景信息和状态文字（包含行走状态提示）
- 右下角提供快捷操作按钮（前往当前、快速到达、查看剧情）

**从 Game Scene 进入：**
- 在世界状态面板（WorldPulseBoard）中新增「进入世界场景 →」按钮，点击可跳转到 `/world/field`

- `@react-three/fiber@8` — React 声明式 Three.js 渲染器
- `@react-three/drei@9` — R3F 辅助工具集

**组件目录：** `src/components/world3d/`

| 文件 | 职责 |
|------|------|
| `World3DStage.jsx` | 3D Canvas 总容器，组装场景、灯光、雾效 |
| `WorldMap.jsx` | 地图渲染：地面平面 + 建筑盒子（fallback）+ 点击标记 |
| `PlayerCharacter.jsx` | 角色控制：fallback 胶囊体模型、A* 寻路移动、走路/待机动画 |
| `BuildingMarker.jsx` | 建筑可点击区域：隐形射线检测 mesh + 浮动光环 |
| `CameraRig.jsx` | 2.5D 侧面相机：平滑跟随角色 X 位置 |
| `pathfinding.js` | A* 网格寻路：世界坐标 ↔ 网格坐标转换、障碍物注册、路径简化 |

**交互流程（3D 版）：**
```
点击建筑/地面 → Raycasting 命中
  → 建筑: 获取建筑前方坐标作为目标
  → 地面: 以命中点作为目标
  → A* 寻路计算避开建筑的路径
  → 角色沿路径移动 + 播放 walk 动画（上下摆动）
  → 到达目标 → onArrived 回调
  → React 弹出 VN 风格剧情面板
```

**素材替换指南：**

当前使用程序化生成的 fallback 几何体。接入真实素材时：
1. **角色 OBJ 模型**：在 `PlayerCharacter.jsx` 中使用 `useLoader(OBJLoader, url)` 加载，替换 `characterMesh`
2. **地图 OBJ 模型**：在 `WorldMap.jsx` 中加载，替换 colored boxes
3. **素材路径约定**：
   - 角色：`public/assets/avatars/player.obj` + `.mtl`
   - 地图：`public/assets/world/{metro|town|village}/map.obj` + `.mtl`
4. **动画说明**：OBJ 格式不支持骨骼动画，后续如需行走/待机动画需改用 GLTF/GLB 格式（带 AnimationClip），或使用 FBX + FBXLoader

## 8. 接下来要完善的核心方向

为了把它真正做成“模拟真实社会运行机制”的人生游戏，建议按下面顺序推进。

### 8.1 第一阶段：把世界状态后端化

优先新增后端持久化模型：

- `player_world_state`
- `player_event`
- `player_action_log`
- `player_schedule`

目标：

- 所有金钱、健康、情绪、关系、时间都由后端保存
- 前端只负责显示，不再自己维护真状态

### 8.2 第二阶段：建立事件引擎

把现实事件从“写死在前端”变成“后端根据状态生成”。

事件类型建议优先做：

- 房租/账单
- 睡眠透支
- 家庭压力
- 求职/工作绩效
- 社交疏离
- 偶发机会

每个事件都需要：

- 触发条件
- 紧急度
- 截止时间
- 可选动作
- 立即后果
- 延迟后果

### 8.3 第三阶段：把主动推送改为状态驱动

主动推送不应该再以“时间桶随机关心”作为主体，而应变成：

- 有事件时才推
- 推送优先级由世界状态决定
- NPC 只是事件表达者，不是事件决策者

理想链路：

`世界状态 -> 事件引擎 -> 推送路由 -> NPC 叙事润色 -> 通知/聊天/剧情入口`

### 8.4 第四阶段：扩展 NPC 社会网络

当前只有“虚拟母亲”一个 NPC，适合测试，不适合构建社会模拟。

下一批建议 NPC：

- 父亲
- 朋友
- 同事/上司
- 房东
- 恋爱对象或潜在人际对象
- 学校/机构/平台型系统角色

### 8.5 第五阶段：前端升级为“人生操作台”

前端可继续扩展为以下模块：

- 世界状态总览面板
- 时间推进器
- 现实事件收件箱
- 剧情过场播放器
- 关系网络面板
- 金钱流动面板
- 健康/情绪波动面板
- 重要人生节点回放

## 9. 当前设计结论

当前项目已经从“AI 聊天壳子”开始过渡到“人生模拟游戏框架”，但离“真实社会模拟”还有关键一步：

要把核心驱动力从“随机聊天”切换为“状态、事件、关系、时间、资源”。

也就是：

- AI 不负责决定世界
- 世界机制先决定发生什么
- AI 负责把这些事说得像真实的人

这是后续整个系统架构最重要的原则。
